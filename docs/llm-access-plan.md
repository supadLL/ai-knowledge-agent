# LLM Access Plan

## Goal

Add a local model-access layer to AI Knowledge Agent so knowledge-base operations can use:

- Direct OpenAI-compatible API keys.
- Upstream relay services such as sub2api, used only as account providers inside this app.
- Codex OAuth / token JSON import, saved into the local Fuel Pool with masked API responses.
- Internal-only account routing for RAG operations without changing local Codex or other API proxy settings.

This project should not copy the full cockpit-tools Rust gateway into the Python MVP. Instead, it should expose a smaller account registry, provider selector, health test, and usage tracker that fit the current FastAPI and static UI architecture.

## Current Implementation Moved Forward

The account import flow is now treated as current work, not a later-only idea.

Internal routing boundary:

- Imported account `base_url`: internal upstream target used by Fuel Pool. It is not the local address that other tools should call.
- Fuel Pool is only used by AI Knowledge Agent's own RAG/eval flows.
- The app does not expose a general-purpose local API relay for other tools.
- The app does not replace, reconfigure, or require any existing local reverse proxy service.
- The OAuth callback listener is temporary auth plumbing only; it is not a model relay endpoint.

### OAuth Login

Backend module: `src/ai_knowledge_agent/codex_oauth.py`

Flow:

1. `POST /api/oauth/codex/start` creates a Codex OAuth login session.
2. The server generates a PKCE `code_verifier`, `code_challenge`, `state`, and `login_id`.
3. The authorization URL uses the same Codex OAuth shape as cockpit-tools:
   - `client_id`: `app_EMoamEEZ73f0CkXaXp7hrann`
   - `redirect_uri`: `http://localhost:1455/auth/callback`
   - `scope`: `openid profile email offline_access api.connectors.read api.connectors.invoke`
   - `originator`: `codex_vscode`
   - `code_challenge_method`: `S256`
4. The app tries to start a local callback listener on `127.0.0.1:1455`.
5. If the port is unavailable, the same UI supports manually pasting the callback URL.
6. `POST /api/oauth/codex/callback-url` verifies the pasted callback `state` and stores the code.
7. `POST /api/oauth/codex/complete` exchanges the code at `https://auth.openai.com/oauth/token`.
8. The resulting `access_token` is saved as an enabled `codex_local_access` Fuel Pool account with base URL `https://chatgpt.com/backend-api/codex`.
9. API responses only return masked token previews, never raw upstream tokens.

Important boundary: OAuth imports the account credential and the Python Codex proxy now translates non-streaming requests generated by AI Knowledge Agent's internal RAG flow into upstream `responses` calls. It is not exposed as a general local API endpoint for external tools. Streaming, WebSocket responses, images, and advanced account routing are still follow-up work.

### Internal Reverse Proxy

Backend module: `src/ai_knowledge_agent/codex_proxy.py`

Current flow:

1. The RAG answer flow asks Fuel Pool for a model completion.
2. If the selected account is `codex_local_access`, Fuel Pool routes through `forward_codex_chat_completion`.
3. The proxy converts chat messages into a `responses` request body:
   - `system` role becomes `developer`.
   - text messages become `input_text` / `output_text` content parts.
   - tool calls and tool outputs are mapped into responses-compatible items.
   - `stream` is forced upstream while local streaming remains disabled for now.
4. The proxy sends the request to `https://chatgpt.com/backend-api/codex/responses` with `Authorization: Bearer <access_token>`.
5. JSON or SSE upstream payloads are parsed.
6. The response is converted back into OpenAI-compatible `chat.completion` shape with `choices[0].message.content` and normalized usage fields.
7. Fuel Pool records latency, success/failure, token usage, and account cooldown state.

Current supported scope:

- Non-streaming text chat completions.
- RAG ask flow through Fuel Pool.
- Masked account storage and redacted failure recording.
- Windows DPAPI-backed local secret payloads for refresh credentials.
- Refresh-token exchange before Codex account generation when refresh credentials are available.

Follow-up scope:

- Streaming SSE conversion for local clients.
- Images and multipart image edit/generation calls.
- WebSocket responses.
- Per-account upstream quota and health probing.
- Account-id extraction and explicit upstream account selection headers.

### Token / JSON Import

The Token / J tab accepts pasted session JSON, `auth.json`, account JSON, Sub2API JSON, raw access tokens, or `accessToken` / `access_token` fields. The parser extracts a token and creates a `codex_local_access` Fuel Pool account.

### API Key Import

The API Key tab offers provider presets for upstream relay services, OpenAI-compatible providers, and common API gateways. The submitted account is persisted to `llm_accounts` and displayed with a masked key.

### Remaining Follow-Up

- Store `refresh_token` / `id_token` in a dedicated encrypted or redacted local secret payload instead of only keeping the access token in the MVP `api_key` field.
- Add token refresh before provider calls when the Codex token expires.
- Add true upstream quota display only if the selected gateway/provider exposes quota information.

## Provider Types

### 1. OpenAI-Compatible API

Fields:

- `name`
- `base_url`
- `api_key`
- `model`
- `enabled`

This covers OpenAI, compatible commercial providers, self-hosted gateways, and many relay services.

### 2. Upstream Relay / sub2api

Same wire format as OpenAI-compatible, but labeled separately in the UI because users may provide a relay account as an upstream provider for this app only.

Suggested fields:

- `base_url`: user provided; do not auto-fill a localhost port unless a service is known to be running there
- `model`: user provided
- `api_key`: optional if the relay does not require one

### 3. Codex Account Relay

Treat imported Codex credentials as an internal upstream account for this app's RAG work.

Suggested fields:

- `base_url`: upstream Codex API URL, defaulting to `https://chatgpt.com/backend-api/codex`
- `api_key`: imported access token or compatible key
- `model`: user supplied

AI Knowledge Agent calls this only from its own Fuel Pool execution path. It should not write or modify local Codex configuration.

### 4. Auth / Session Token Import

Current scope:

- OAuth login is supported through a local PKCE flow.
- Manual callback paste is supported when the local callback port is unavailable.
- Pasted token/session JSON import is supported for local Fuel Pool account creation.
- API responses are masked and diagnostics redact known account secrets.

Current hardening:

- `refresh_token`, `id_token`, expiry, and token type metadata are stored in a dedicated local credential payload.
- On Windows, that payload is protected with the current user's DPAPI before it is written to `app.db`.
- Fuel Pool refreshes Codex access tokens before app-internal generation when refresh credentials exist.

Hardening still needed:

- Token invalidation flow.
- Clear per-account expiry display.

## Generation Integration

Current generation flow reads environment variables through `AppConfig`.

Implementation path:

1. Keep environment-based provider config as fallback.
2. Add `LlmAccountStore` and `LlmAccountService`.
3. When an enabled active account exists, construct `OpenAICompatibleChatGenerator` from that account.
4. Capture provider latency and usage in `llm_usage_events`.
5. Return `generation_provider`, `model`, and tracked usage metadata from `/api/ask`.

## Safety Rules

- Never return raw secrets from API responses.
- Never print secrets in diagnostics, logs, docs, or test output.
- Mask keys as `sk-...abcd` style where possible.
- Treat OAuth/session imports as local-only account credentials and keep responses masked.
- Move refresh tokens into a dedicated encrypted or redacted secret payload before enabling automatic refresh.
- Default any local gateway guidance to `127.0.0.1`.
- Do not promise true account quota unless the upstream provider exposes quota information.
